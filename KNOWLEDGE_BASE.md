## Анимация "Живой контур" (Interactive Border Sparkle)

Этот паттерн используется для создания движущегося светового контура вокруг элементов сложной формы (овалов, иконок).

### Ключевые компоненты:

1. **CSS (index.css):**
   ```css
   .sparkle-border-mask {
     -webkit-mask: linear-gradient(#fff 0 0) content-box, linear-gradient(#fff 0 0);
     mask: linear-gradient(#fff 0 0) content-box, linear-gradient(#fff 0 0);
     -webkit-mask-composite: xor;
     mask-composite: exclude;
     padding: 3px; /* Параметр для настройки толщины контура */
   }
   ```
   *Суть: Метод `mask-composite: exclude` вырезает внутреннюю часть элемента, оставляя лишь маску толщиной `padding` по периметру.*

2. **React (Home.tsx):**
   - Используется `motion.div` (из `motion/react`) для плавного вращения (`animate={{ rotate: 360 }}`).
   - Градиент: `conic-gradient` создает световой "пучок" (beam).
   - Размещение: Анимируемый элемент должен иметь размер `inset-[-150%]`, чтобы пучок описывал полный круг внутри маски.

## Автоматический деплой на GitHub Pages (PWA)

Для автоматизации сборки и деплоя приложения (включая ресурсы PWA) используется GitHub Actions.

### Конфигурация Workflow (`.github/workflows/deploy.yml`):

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: ["main"]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }} # Важно: использование точки, а не двоеточия
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci # Рекомендуется использовать npm ci для воспроизводимых сборок в CI
      - name: Build
        run: npm run build
      - name: Setup Pages
        uses: actions/configure-pages@v4
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: './dist'
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

### Критические нюансы:

1.  **Права доступа (Permissions):** Обязательно наличие `pages: write` и `id-token: write` для успешной публикации.
2.  **Синтаксис переменных:** Обращение к выходам шага (`outputs`) выполняется через точку: `steps.id.outputs.key`. Использование двоеточия (`steps:id`) является ошибочным и вызывает сбой парсинга workflow.
3.  **Маршрутизация (HashRouter):** Для GitHub Pages настоятельно рекомендуется использовать `HashRouter` вместо `BrowserRouter`, так как GitHub не умеет перенаправлять запросы на `index.html` при прямых заходах по путям (например, при перезагрузке на `/temple` будет 404). С `HashRouter` пути выглядят как `/#/temple`, и они всегда работают.
4.  **Относительные пути (base: './'):** При использовании `HashRouter` можно установить `base: './'` в `vite.config.ts`, что делает приложение переносимым между любыми папками и поддоменами без необходимости жестко прописывать नाम репозитория.
5.  **Путь к артефактам:** Для сборки на Vite путь всегда `./dist`.
6.  **Среда выполнения:** Использование `npm ci` гарантирует, что `package-lock.json` будет соблюден в точности.

## 3. Принудительное окно установки (Installation UI)

Так как браузеры (особенно Safari на iOS) не всегда предлагают установку корректно, мы используем кастомное окно установки.

### Логика работы (`src/components/InstallPrompt.tsx`):

1.  **Определение платформы:**
    -   **Android/Chrome:** Слушаем событие `beforeinstallprompt`. Сохраняем событие и показываем кнопку "Установить".
    -   **iOS/Safari:** Проверяем `navigator.userAgent` (наличие iPhone/iPad) и отсутствие `window.navigator.standalone`.
    -   **Уже установлено:** Проверяем медиа-запрос `(display-mode: standalone)` — если приложение уже запущено как PWA, окно не показывается.

2.  **Поведение для iOS:**
    На iOS установка возможна только через нативное меню "Поделиться". В интерфейсе мы показываем пошаговую инструкцию с иконками:
    -   Иконка "Поделиться" (Share).
    -   Пункт «На экран "Домой"» (Add to Home Screen).

3.  **Управление состоянием:**
    -   Используем `localStorage` (например, `iosPwaPrompted`), чтобы не показывать окно при каждом визите после того, как пользователь его закрыл.
    -   Используем `AnimatePresence` и `motion` для плавного появления/исчезновения баннера.

4.  **Стилизация:**
    -   Фиксированное позиционирование (`fixed bottom-24`) над навигационной панелью.
    -   Использование переменных проекта (`var(--color-cinnabar)`, `var(--color-parchment)`).

## 4. Механизм резервного копирования и восстановления бинарных ассетов

Искусственный интеллект (AI Studio агент) часто испытывает трудности с загрузкой, редактированием или сохранением больших и бинарных файлов (шрифты, изображения, книги `.epub`/`.fb2`). Для решения проблем потери файлов или их повреждения реализован специальный механизм (бекап/рестор).

### Суть механизма:

Мы используем скрипты на Node.js для обхода директории `public/`, кодирования всех бинарных файлов в формат Base64 и сохранения их в текстовые файлы в безопасной директории `scripts/assets_backup/`. Затем, при необходимости или перед сборкой, скрипт восстановления декодирует Base64 и воссоздает файлы в исходных папках.

### Реализация:

1.  **Создание резервных копий (scripts/backup-assets.cjs):**
    - Скрипт рекурсивно обходит папку `public/`.
    - Ищет файлы с бинарными расширениями: `.png`, `.jpg`, `.jpeg`, `.webp`, `.svg`, `.gif`, `.ico`, `.ttf`, `.woff`, `.woff2`, `.epub`, `.fb2`.
    - Относительный путь файла (`relativePath`) энкодится. Символы слэшей (`/` или `\`) заменяются на маркер `__PATH__`.
    - Содержимое файла читается как буфер и конвертируется в строку: `buffer.toString('base64')`. 
    - Строка сохраняется в виде текстового файла с расширением `.base64.txt` в папку `scripts/assets_backup/` (пример: `images__PATH__Fon_glav.webp.base64.txt`).

2.  **Восстановление из резервных копий (scripts/restore-assets.cjs):**
    - Скрипт читает содержимое папки `scripts/assets_backup/`.
    - Находит все файлы `.base64.txt`.
    - Декодирует имена файлов (заменяет `__PATH__` обратно на `/` для получения оригинальных путей внутри папки `public/`).
    - Создает необходимые поддиректории.
    - Декодирует Base64 через `Buffer.from(cleanBase64, 'base64')` и записывает бинарный файл по исходному пути.

### Преимущества подхода:
- **Сохранение целостности:** Текстовый код Base64 легко считывается ИИ-ассистентом и текстовым редактором, без непредсказуемого изменения байтов в файлах.
- **Портативность:** Можно легко удалить испорченные дубликаты (с постфиксами вроде `-1`), восстановив первозданные файлы в один клик командой `npx -y tsx scripts/restore-assets.cjs`.
- **Интеграция:** Резервирование (`backup-assets.cjs`) происходит по мере добавления файлов, а восстановление (`restore-assets.cjs`) можно запускать как часть сборочных скриптов проекта.

## 5. Полная настройка PWA (опыт проектов "Как росли детки" и "Метанойя")

Для того чтобы приложение можно было установить на телефоне, требуется правильно связать Manifest, Service Worker и файлы иконок.

### Важные составляющие:
1. **Иконки (`icon_192.png`, `icon_512.png`):** Должны физически лежать в корне папки `public/` (а не в `src/` или подпапке `public/fonts/`). Это важно для того, чтобы `manifest.json` и `sw.js` находили их по путям `./icon_192.png`.
2. **`manifest.json`:**
   - `"start_url": "./"` — точка в пути сохраняет маршурт при заходе из подпапки на GitHub Pages.
   - Обязательно укажите `"display": "standalone"`.
3. **`pwa-setup.js`:** Скрипт регистрации кладем также в `public/` и вызываем в `index.html`. Путь к SW должен быть относительным, без слэша в начале: `navigator.serviceWorker.register('sw.js')`.
4. **`sw.js` (Кэширование и Offline Fallback):**
   - Кешируемые пути массива `urlsToCache` должны начинаться с текущей директории: `'./', 'index.html', 'manifest.json', 'icon_192.png'`.
   - В событии `fetch`, в случае перехвата роутинга от React Router (`event.request.mode === 'navigate'`), нужно делать Fallback на `'./'`, а не на `/index.html`: `return caches.match('./');`.

## 6. Всплывающее окно успешного обновления (UpdatePopup)

Чтобы пользователи PWA узнавали о новых функциях после фонового обновления (или очистки кэша сборок), используется паттерн проверки версии через `localStorage`.

### Логика компонента (`UpdatePopup.tsx`):
1. В коде объявляется константа версии: `const CURRENT_VERSION = '1.0.1';`.
2. В `useEffect` происходит попытка чтения `appVersion` из `localStorage`.
3. Сценарии:
   - Если версии нет (первый заход после ревизии) — молча записываем `CURRENT_VERSION` в хранилище, окно не показываем.
   - Если версия в `localStorage` не совпадает с `CURRENT_VERSION` — приложение обновилось. Вызываем `setShow(true)`, показывая пользователю модальное окно.
4. При закрытии (или клике "Отлично") — записываем текущую версию в `localStorage` и скрываем окно.

Этот опыт позволяет просто и нативно оповещать о новых фичах, достаточно лишь инкрементировать номер версии в коде.
